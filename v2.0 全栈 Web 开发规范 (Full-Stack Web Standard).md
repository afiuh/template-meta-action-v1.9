## 2. 技术栈选型规则 (Stack Selection Rules)

AI 必须按顺序匹配以下规则，命中即停，不得随意更改。

### 2.1 后端框架
1.  **数据科学/AI/爬虫：** 强制 `FastAPI` (Python)。
2.  **Python 偏好：** 高性能/实时选 `FastAPI`，否则 `Flask`。
3.  **JS/TS 偏好：** 高性能选 `Fastify`，否则 `Express`。
4.  **Go 偏好：** 强制 `Gin`。
5.  **其他：** Java(`Spring Boot`), PHP(`Laravel`), Ruby(`Rails`)。
6.  **默认：** `FastAPI`。

### 2.2 数据库与 ORM (若 database_needed=true)
1.  **Python:** 高性能 `SQLAlchemy`，异步 `Tortoise ORM`。
2.  **JS/TS:** 强制 `Prisma` (类型安全)。
3.  **Go:** 强制 `GORM`。
4.  **默认：** SQLite (开发) + PostgreSQL (生产)。

### 2.3 前端状态管理
1.  **简单应用：** 框架原生 (React Context / Vue Reactive)。
2.  **复杂应用：** `Zustand` (React) 或 `Pinia` (Vue)。
3.  **服务端状态：** 强制 `React Query` 或 `SWR` (管理 API 缓存与同步)。

## 4. API 契约与安全规范 (API Contract & Security)

### 4.1 统一响应格式
所有接口必须返回以下 JSON 结构：
```json
{
  "code": 0,          // 0=成功，非 0=业务错误
  "message": "success",
  "data": null        // 具体数据
}
```

### 4.2 HTTP 状态码规范
-  **成功：** HTTP 200 + 业务 code: 0。
-  **客户端错误：** HTTP 400/401/403/404 + 业务 code: 4xx。
-  **服务端错误：** HTTP 500 + 业务 code: 5xx (前端提示"系统繁忙")。
-  **前端拦截：** 必须封装 Axios 拦截器，统一处理 HTTP 错误与业务 code 错误。

### 4.3 安全强制项
-  **CORS：** 后端必须配置跨域中间件，仅允许指定域名。
-  **鉴权：** 所有写操作 (POST/PUT/DELETE) 必须验证 Token (JWT/Session)。
-  **输入验证：** 后端必须二次验证所有前端输入 (Pydantic/Zod)，禁止信任前端。
-  **敏感信息：** 禁止将 Secret/Key 硬编码，必须使用 `.env`。

## 5. v2.0 体系集成规范 (v2.0 Integration)

### 5.1 元动作注释 (Meta-Action Tags)
所有代码块必须标注元动作标签，格式如下：
-  **JS/TS:** `// [XX 动作名称]`
-  **Python:** `# [XX 动作名称]`

**关键映射点：**
-  **前端 API 调用：** `// [I16 通信] 请求后端接口`
-  **后端路由接收：** `# [I14 用户输入] 接收前端请求`
-  **数据库操作：** `# [I15 存储] 读写数据库`
-  **异常处理：** `# [F12 捕获] 捕获业务异常`

### 5.2 SOP 流程遵守
本项目开发必须严格遵守 v1.9/v2.0 SOP 四阶段：
1.  **架构设计：** 基于本规范选型，输出文件清单。
2.  **分层拆解：** 每个文件独立拆解。
3.  **语义蓝图：** 生成元动作伪代码，AI 自检通过后冻结。
4.  **代码实现：** 严格按蓝图编码，每个文件完成后 Git 提交。

### 5.3 知识沉淀
-  **决策记录：** 项目初始化时自动生成 `docs/DECISIONS.md`。
-  **经验卡片：** 遇到 Bug 修正后，必须生成经验卡片存入 `docs/experience/01-runtime/`。
-  **技术债务：** 临时方案必须记录到 `docs/TECH_DEBT.md`。

## 6. AI 执行流程 (AI Execution Flow)

1.  **收集参数：** 若用户未提供，主动提问获取 Section 1 参数。
2.  **技术选型：** 按 Section 2 规则选择框架，写入 `DECISIONS.md`。
3.  **生成结构：** 按 Section 3 生成目录树。
4.  **进入 SOP：** 引导用户进入 v1.9 SOP 阶段一 (架构设计)。
5.  **循环开发：** 按文件循环执行 SOP 阶段二至四。
6.  **最终交付：** 确保所有代码含元动作注释，所有接口符合 Section 4 规范。
