## 1. 体系概述

### 1.1 核心理念
本体系旨在通过**语义化标注**和**经验工程化**，将开发过程中的隐性知识显性化，实现：
*   **概率空间最小化：** 通过规则约束 AI 生成代码的随机性。
*   **风险显性化：** 通过元动作标注暴露潜在风险点（如 IO、网络、存储）。
*   **持续迭代：** 通过经验卡片机制，让项目越做越聪明。

### 1.2 核心组件
| 组件 | 文件/目录 | 作用 |
| :--- | :--- | :--- |
| **AI 规则** | `.cursorrules` | 约束 AI 行为，强制元动作标注 |
| **经验库** | `docs/experience/` | 存储种子经验 (`00-seed`) 和运行时经验 (`01-runtime`) |
| **自动化** | `.husky/`, `scripts/` | 提交前检查、经验晋升脚本 |
| **配置** | `.eslintrc`, `.flake8` | 多语言代码质量检查 |
| **上下文隔离** | `.cursorignore` | 排除无关文件，优化 AI 索引性能 |

## 2. 快速开始

### 2.1 前提条件
| 工具 | 版本 | 验证命令 |
| :--- | :--- | :--- |
| **Node.js** | v18+ | `node -v` |
| **Python** | v3.8+ | `python --version` |
| **Git** | 最新版 | `git --version` |
| **Cursor** | 最新版 | - |

### 2.2 创建新项目 (推荐方式)
1.  打开模板仓库页面：https://github.com/afiuh/template-meta-action-v1.9
2.  点击绿色按钮 **"Use this template"** → **"Create a new repository"**。
3.  输入新项目名称，点击 **Create repository**。
4.  克隆新项目到本地：
    ```bash
    git clone https://github.com/你的用户名/<项目名>.git
    cd <项目名>
    npm install
    ```

### 2.3 项目定制化 (必须)
GitHub 已自动初始化 Git 历史，但需更新项目元数据以确立主权。

#### 1. 更新项目身份信息
修改以下文件以匹配新项目：
*   **`package.json`**: 修改 `"name"` 字段为你的项目名。
*   **`README.md`**: 修改标题和项目描述。

#### 2. 配置上游仓库 (推荐)
为了未来能同步模板更新，建议建立只读连接：
```bash
git remote add upstream https://github.com/afiuh/template-meta-action-v1.9.git
```

#### 3. 提交定制变更
```bash
git add .
git commit -m "chore: 定制项目元数据为 [项目名称]"
git push -u origin main
```

## 3. 开发工作流 (核心方法论)

本体系采用 **四阶段标准作业流程 (SOP)**，核心模型为 **"全局架构规划 + 文件级迭代实现"**。

### 3.1 虚拟"AI 阅读文件夹”流程 (性能优化)

为了优化 AI 索引性能，避免扫描无关文件，本体系采用 **上下文隔离** 策略。

#### 原理
| 传统方式 | **优化后 (虚拟 AI 阅读文件夹)** |
| :--- | :--- |
| AI 扫描所有文件 (Worktree 模式) | **AI 只读你指定的文件** (Local 模式) |
| 容易卡顿 (索引大量无关文件) | **速度极快** (精准控制上下文) |
| 上下文污染 (读到无关文档) | **纯净上下文** (只读核心规则) |

#### 配置说明

**`.cursorignore` 已预配置排除以下文件：**
```text
# 排除文档目录 (AI 默认不读，除非用 @ 引用)
docs/**/*.md
README.md
MANUAL.md
PROMPT_TEMPLATES.md

# 排除依赖和构建目录
node_modules/
dist/
build/
.git/
```

**效果：**
*   AI **不会自动索引** 这些文件 (大幅提升速度)。
*   你仍然可以用 **`@` 符号** 显式引用需要的文件。
*   实现了"虚拟 AI 阅读文件夹"的效果，无需物理移动文件。

#### 标准启动指令

**每次新建 Chat 后，使用以下指令启动 SOP 流程：**

```markdown
@.cursorrules @docs/experience/00-seed/

我要实现 [功能名称]。

```

**关键点：**
*   **`@.cursorrules`** - 强制 AI 读取规则文件。
*   **`@docs/experience/00-seed/`** - 强制 AI 读取经验卡片。

#### Cursor 设置推荐

| 设置项 | 推荐值 | 说明 |
| :--- | :--- | :--- |
| **模式** | `∞ Agent` | 支持多步骤任务和文件读写 |
| **上下文范围** | `Local` | 只读当前文件 + `@` 引用的文件 (速度快) |
| **模型** | `Claude 3.5 Sonnet` | 对指令遵循最好 |

### 3.2 流程概览 (迭代模型)

```text
[阶段一：架构设计] (全局一次)
       ↓
   生成文件清单
       ↓
   ┌──────────────────────────────────────┐
   │  循环：对清单中的每个文件执行        │
   │                                      │
   │  [阶段二：分层拆解] (针对当前文件)    │
   │         ↓                            │
   │  [阶段三：语义蓝图生成] (针对当前文件) │
   │         ↓                            │
   │  [阶段四：代码实现与验证] (针对当前文件)│
   │                                      │
   └──────────────────────────────────────┘
       ↓
   所有文件完成 → 项目功能完成
```

### 3.3 阶段一：架构设计 (Architecture Design)
**目标：** 明确系统边界，生成文件清单。  
**执行频率：** 每个功能/需求 **一次性**。

1.  **需求澄清：** 人类提出功能目标，AI 通过提问澄清细节。
2.  **结构提案：** AI 提出多文件、多模块的代码结构方案，说明依赖关系与编写顺序。
3.  **人类确认：** 人类审核结构合理性，生成 **文件清单**。

### 3.4 阶段二至四：文件级迭代 (File-Level Iteration)
**目标：** 逐个文件高质量落地。  
**执行频率：** 对架构设计中的 **每个文件** 循环执行。

#### 阶段二：分层拆解 (Hierarchical Decomposition)
1.  **文件定义：** AI 说明 **当前文件** 的职责边界。
2.  **递归细分：** 每次只细分一级，人类确认后再进入下一级。
3.  **终止条件：** 直到每个细分单元都能轻松映射为 3-5 个元动作。

#### 阶段三：语义蓝图生成 (Semantic Blueprint Generation)
1.  **元动作映射：** AI 将 **当前文件** 的最底层单元转换为元动作序列。
2.  **生成元动作文档：** 创建不含实际代码的伪代码文档。
3.  **人类审查 (只看风险)：**
    *   **必查：** `I15 存储 `, `I16 通信`, `F12 捕获` 是否成对出现。
    *   **选查：** 业务逻辑核心分支 (`C6 条件`)。
    *   **跳过：** 常规渲染 (`I13`)、常规转换 (`M5`)。
4.  **蓝图冻结：** 确认后，该文档成为 **当前文件** 代码生成的唯一依据。

#### 阶段四：代码实现与验证 (Implementation & Verification)
1.  **代码生成：** AI 严格按照 **当前文件的元动作文档** 编写代码。
2.  **人机协同审查：** 执行自动检查 (ESLint/Husky)。
3.  **经验生成：** 若发生修正，AI 自动生成经验卡片。
4.  **进入下一文件：** 返回阶段二，处理清单中的下一个文件。

### 3.5 关键控制点

| 控制点 | 范围 | 说明 | 目的 |
| :--- | :--- | :--- | :--- |
| **文件清单** | 全局 | 阶段一必须输出明确的文件列表 | 防止遗漏模块，明确迭代范围 |
| **蓝图冻结** | 文件级 | 每个文件代码生成前必须冻结蓝图 | 防止代码偏离逻辑，减少返工 |
| **逐个文件** | 文件级 | 严禁一次性生成所有文件代码 | 控制 AI 上下文，保证每个文件质量 |
| **经验沉淀** | 文件级 | 每个文件完成后检查是否有新经验 | 确保知识持续积累 |
| **上下文隔离** | 全局 | 使用 `.cursorignore` + Local 模式 | 优化性能，避免索引无关文件 |

## 4. 经验卡片系统

### 4.1 卡片分类
| 类型 | 目录 | 来源 | 更新方式 |
| :--- | :--- | :--- | :--- |
| **种子经验** | `docs/experience/00-seed/` | 模板继承 | 从 `upstream` 同步 |
| **运行时经验** | `docs/experience/01-runtime/` | 项目自产 | 本地积累，可晋升 |

### 4.2 创建经验卡片
当遇到 AI 重复犯错或发现新风险时：
1.  修正代码。
2.  在 Cursor 中输入：
    ```
    请根据刚才的修正，生成一张经验卡片。
    格式遵循 docs/experience/TEMPLATE.md (如有)。
    包含 YAML Frontmatter (id, meta-actions, enforce-type)。
    保存到 docs/experience/01-runtime/ 目录。
    ```
3.  提交卡片文件。

### 4.3 经验晋升 (贡献回模板)
当某张运行时经验被验证为通用时：
1.  **AI 泛化：** 让 AI 去除业务特定逻辑，保留通用规则。
2.  **执行脚本：**
    ```bash
    npm run promote <卡片 ID>
    ```
3.  **提交 PR：** 脚本会自动创建分支，推送到 `upstream` 仓库并创建 Pull Request。
4.  **审核：** 模板维护者审核通过后，合并入 `00-seed/`。

## 6. 多语言支持

### 6.1 JavaScript / TypeScript
*   **Lint 工具：** ESLint
*   **元动作注释：** `// [I15 存储]`
*   **约束：** `localStorage` 必须 try-catch。

### 6.2 Python
*   **Lint 工具：** Flake8, Black
*   **元动作注释：** `# [I15 存储]`
*   **约束：** 文件 IO 必须用 `with`，禁止裸 `except:`。

### 6.3 元动作映射表
| 元动作 | JavaScript | Python |
| :--- | :--- | :--- |
| **[I15 存储]** | `localStorage` | `open()` / `json.dump()` |
| **[I16 通信]** | `fetch()` | `requests.get()` |
| **[F12 捕获]** | `try-catch` | `try-except` |
| **[I13 渲染]** | `DOM 操作` | `print()` / `logging` |

## 7. 常见问题 (FAQ)

### 7.1 AI 性能问题
| 问题 | 解决方案 |
| :--- | :--- |
| **AI 响应慢/卡死** | 确认使用 `Local` 模式，检查 `.cursorignore` 是否排除 `node_modules/` |
| **AI 不遵守 SOP 流程** | 新建 Chat，使用 `@.cursorrules @docs/experience/00-seed/` 显式引用 |
| **AI 直接生成代码** | 在对话中提醒"请严格遵守阶段一，先提问澄清需求" |

### 7.2 Git Hook 问题
| 问题 | 解决方案 |
| :--- | :--- |
| **Husky 不执行** | 检查 `.husky/pre-commit` 换行符是否为 **LF** |
| **提交被拦截** | 运行 `npx lint-staged --fix` 修复后重试 |

### 7.3 AI 不遵守规则
| 问题 | 解决方案 |
| :--- | :--- |
| **AI 不读经验卡片** | 使用 `@docs/experience/00-seed/` 显式引用 |
| **AI 不标注元动作** | 在对话中提醒"请标注元动作注释" |
